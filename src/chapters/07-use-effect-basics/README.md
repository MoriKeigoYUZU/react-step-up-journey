# Chapter 07: useEffectの基本

## 副作用（Side Effect）ってなに？

Reactコンポーネントの本来の仕事は **「propsやstateをもとに画面を描画すること」** です。
でも現実のアプリケーションでは、描画以外にも色々なことをする必要がありますよね。

- ページのタイトルを変える
- タイマーを動かす
- サーバーからデータを取得する
- ブラウザのイベントを監視する

こういった **「描画以外の処理」** のことを **副作用（Side Effect）** と呼びます。

### アナロジー: 料理と片付け

こんな風に考えてみてください。

- **料理を作る（レンダリング）**: 材料（props, state）をもとに、お皿に盛り付ける（画面を描画する）
- **片付けをする（副作用）**: 料理が終わった後に、キッチンを掃除したり、残った材料を冷蔵庫にしまう

料理（レンダリング）が本業ですが、片付け（副作用）も必要ですよね。
`useEffect` は、**「料理が終わった後に片付けをするためのツール」** です。

---

## useEffectの基本的な使い方

```tsx
import { useEffect } from 'react';

function MyComponent() {
  useEffect(() => {
    // ここに「副作用」の処理を書く
    console.log('画面が描画された後に実行されるよ');
  });

  return <div>コンポーネント</div>;
}
```

**大事なポイント**: `useEffect` に渡した関数は、**画面の描画が完了した後に** 実行されます。

つまり、以下の順番で処理が進みます:

```
1. コンポーネントの関数が実行される
2. JSXをもとに画面が描画される（ブラウザに表示される）
3. useEffect の中の処理が実行される  ← ここ！
```

### なぜ「描画の後」なのか？

描画をブロックしないためです。

もし `useEffect` の処理が重い場合（サーバーからデータを取得するなど）、
描画の前に実行してしまうと、ユーザーは画面が真っ白な状態で待たされてしまいます。

先に画面を表示してから、裏で処理を進める。これが `useEffect` の設計思想です。

## 依存配列（Dependency Array）

`useEffect` の **第2引数** に渡す配列を **依存配列** と呼びます。
この配列が、「いつ副作用を再実行するか」を決めます。

### パターン1: 依存配列なし（毎回実行）

```tsx
useEffect(() => {
  console.log('毎回のレンダー後に実行される');
});
```

**注意**: これは **ほとんどの場合、間違い** です。
再レンダーのたびに実行されるので、無限ループの原因になりがちです。

### パターン2: 空の配列 `[]`（マウント時のみ）

```tsx
useEffect(() => {
  console.log('最初の1回だけ実行される（マウント時）');
}, []);
```

**「マウント」** とは、コンポーネントが画面に初めて表示されるタイミングのことです。

アナロジーで言えば: **「お店が開店したときだけ行うセットアップ」** です。
テーブルを並べたり、看板を出したり。開店のときだけやって、お客さんが来るたびにはやりませんよね。

### パターン3: 値を入れた配列 `[value]`（値が変わったら実行）

```tsx
useEffect(() => {
  console.log(`countが ${count} に変わりました`);
}, [count]);
```

`count` の値が変わったときだけ実行されます。
別の state が変わっても、`count` が同じなら実行されません。

アナロジー: **「気温が変わったらエアコンの設定を変える」** みたいなもの。
気温が変わったときだけ反応して、それ以外のときは何もしません。

### 依存配列の比較まとめ

```tsx
// パターン1: 依存配列なし → 毎回実行（ほぼ使わない）
useEffect(() => { ... });

// パターン2: 空配列 → マウント時のみ（初期化処理）
useEffect(() => { ... }, []);

// パターン3: 値あり → 値が変わったら実行（値の変化に反応）
useEffect(() => { ... }, [count]);

// パターン3の応用: 複数の値を監視
useEffect(() => { ... }, [count, name]);
```

## クリーンアップ関数: 後片付けの仕組み

`useEffect` の中で **関数を return** すると、それが **クリーンアップ関数** になります。

```tsx
useEffect(() => {
  // セットアップ（副作用の開始）
  const timer = setInterval(() => {
    console.log('tick');
  }, 1000);

  // クリーンアップ（後片付け）
  return () => {
    clearInterval(timer);
  };
}, []);
```

### アナロジー: ホテルのチェックアウト

`useEffect` のクリーンアップは、**ホテルのチェックアウト** に似ています。

- **チェックイン（セットアップ）**: 部屋に入ったら、荷物を広げて、テレビをつける
- **チェックアウト（クリーンアップ）**: 部屋を出るとき、荷物をまとめて、テレビを消す

もしチェックアウトのときに片付けをしなかったら？
テレビはつきっぱなし、水は出しっぱなし...。これが **メモリリーク** です。

### クリーンアップが実行されるタイミング

クリーンアップ関数は **2つのタイミング** で実行されます:

1. **コンポーネントがアンマウントされるとき**（画面から消えるとき）
2. **依存配列の値が変わって、次の副作用が実行される前**

```tsx
useEffect(() => {
  document.title = title;

  return () => {
    document.title = '';  // 次の副作用の前、またはアンマウント時に実行
  };
}, [title]);
```

### なぜクリーンアップが重要なのか

クリーンアップを忘れると、こんな問題が起きます:

- **タイマーの二重起動**: setInterval を clearInterval しないと、タイマーが積み重なる
- **メモリリーク**: イベントリスナーを解除しないと、メモリが解放されない
- **古いデータの更新**: 既にアンマウントされたコンポーネントの state を更新しようとしてエラーになる

## よくある間違いと注意点

### 間違い1: 依存配列を忘れる

```tsx
// NG: 毎レンダーで setInterval が追加される
useEffect(() => {
  setInterval(() => setCount(c => c + 1), 1000);
}); // ← 依存配列がない！

// OK: マウント時に1回だけ setInterval する
useEffect(() => {
  const id = setInterval(() => setCount(c => c + 1), 1000);
  return () => clearInterval(id);
}, []); // ← 空配列で「1回だけ」
```

### 間違い2: 依存配列に必要な値を入れない

```tsx
// NG: message を使っているのに依存配列に入れていない
useEffect(() => {
  console.log(message); // message の古い値が使われる
}, []);

// OK: 使っている値を依存配列に入れる
useEffect(() => {
  console.log(message);
}, [message]);
```

React は **「使っている値は全て依存配列に入れる」** というルールを推奨しています。
ESLint の `react-hooks/exhaustive-deps` ルールが、入れ忘れを教えてくれます。

### 間違い3: useEffect の中で直接 async を使う

```tsx
// NG: useEffect のコールバック自体を async にしない
useEffect(async () => {
  const data = await fetchData();
}, []);

// OK: 中で async 関数を定義して呼ぶ
useEffect(() => {
  const loadData = async () => {
    const data = await fetchData();
  };
  loadData();
}, []);
```

なぜ NG？ `useEffect` のコールバックが返すものは **クリーンアップ関数** でなければなりません。
`async` 関数は `Promise` を返すので、React が混乱してしまいます。

## まとめ

- **useEffect** は「画面の描画が終わった後に実行される処理」を書く場所
- **依存配列** で「いつ再実行するか」を制御する
  - `[]` = マウント時のみ
  - `[value]` = value が変わったとき
  - なし = 毎回（ほぼ使わない）
- **クリーンアップ関数** で後片付けを行う（メモリリーク防止）
- 使っている値は **全て依存配列に入れる**
- useEffect のコールバック自体を `async` にしない
