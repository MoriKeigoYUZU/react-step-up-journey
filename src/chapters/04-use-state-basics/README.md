# Chapter 04: useStateの基本

## State（状態）って何？

ここまでの章では、コンポーネントは **毎回同じもの** を表示していました。
Props を渡せば表示は変わりましたが、「ボタンをクリックしたら変わる」のような **ユーザーの操作に応じた変化** はありませんでした。

ここで登場するのが **State（状態）** です。

### 「ホワイトボード」で考えてみよう

コンポーネントを「教室にいる先生」に例えてみましょう。

- **Props** = 校長先生から渡された「今日の連絡事項」（読むだけ、変更不可）
- **State** = 先生が自分で持っている「ホワイトボード」（自由に書き換えられる）

ホワイトボードに「出席者: 30人」と書いてある。
生徒が1人遅刻してきたら、先生はホワイトボードを消して「出席者: 31人」に書き換える。

State は **コンポーネントが自分で管理するデータ** です。
ユーザーの操作に応じて **書き換え** ることができ、書き換えると **画面が自動で更新** されます。

---

## なぜ普通の変数ではダメなのか

「変数を使えばいいのでは？」と思うかもしれません。
試してみましょう。

```tsx
// これは動かない！
function Counter() {
  let count = 0;

  function handleClick() {
    count = count + 1;
    console.log(count); // 1, 2, 3... と増える
    // でも画面は更新されない！
  }

  return (
    <div>
      <p>カウント: {count}</p>
      <button onClick={handleClick}>+1</button>
    </div>
  );
}
```

なぜ画面が更新されないのでしょうか？

Reactの仕組みを理解する必要があります:
1. Reactは `Counter` **関数を呼び出して** 画面を描画する
2. ボタンがクリックされても、React は **関数を呼び直さない**（普通の変数の場合）
3. たとえ変数の値が変わっても、React は **それを知らない**

つまり、普通の変数を変えても **React に「画面を更新して」と伝えられない** のです。

---

## useState の登場

`useState` は、React に「この値が変わったら画面を更新してね」と伝えるための仕組みです。

```tsx
import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0);
  //     ↑         ↑              ↑
  //   現在の値   更新関数      初期値

  return (
    <div>
      <p>カウント: {count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
    </div>
  );
}
```

### 「箱と更新ボタン」のイメージ

`useState(0)` を呼ぶと、React は:
1. **箱** を用意する（初期値 `0` が入っている）
2. **箱の中身を読む方法**（`count`）を返す
3. **箱の中身を書き換える方法**（`setCount`）を返す

```
useState(0) が返すもの:

  ┌─────────────────┐
  │  箱: [count]    │  ← 現在の値を読める
  │  値: 0          │
  │                 │
  │  ボタン: setCount│  ← これを押すと中身が変わり、画面も更新される
  └─────────────────┘
```

### setCount を呼ぶと何が起きるか

1. `setCount(1)` を呼ぶ
2. React が「あ、state が変わった！」と気づく
3. React が **コンポーネント関数をもう一度呼び出す**（再レンダリング）
4. 今度は `count` が `1` になっている
5. 新しい画面が表示される

```
クリック前:  Counter() → count = 0 → <p>カウント: 0</p>
                         ↓
setCount(1) を呼ぶ
                         ↓
クリック後:  Counter() → count = 1 → <p>カウント: 1</p>
```

---

## なぜ直接変更してはいけないのか

```tsx
// NG: 直接変更
count = count + 1;

// OK: setCount を使う
setCount(count + 1);
```

直接変更すると、React は変更に **気づけません**。
`setCount` を使うことで、React に「値が変わったよ、画面を更新して！」と **通知** できるのです。

手紙を出さずに「連絡したつもり」になっているのと同じです。
ちゃんと手紙（setCount）を出さないと、相手（React）には伝わりません。

---

## 「スナップショット」のメンタルモデル

各レンダリング（関数の呼び出し）は、**そのときの state の写真（スナップショット）** だと考えてください。

```tsx
function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    // このクリックハンドラが実行されるとき、
    // count は「このレンダリング時点での値」
    console.log(count); // 0（最初のレンダリングなら）
    setCount(count + 1);
    console.log(count); // まだ 0！（次のレンダリングまで変わらない）
  }

  return (
    <div>
      <p>カウント: {count}</p>
      <button onClick={handleClick}>+1</button>
    </div>
  );
}
```

重要なポイント:
- `setCount` を呼んでも、**その場で** `count` は変わらない
- 次のレンダリング（関数の再呼び出し）で、新しい `count` が使われる
- 各レンダリングは **独立したスナップショット**

---

## useState のルール

1. **コンポーネントのトップレベルで呼ぶ**（if文やループの中で呼ばない）
2. **React のコンポーネント関数の中でだけ呼ぶ**（普通の関数では使えない）

```tsx
// OK
function Counter() {
  const [count, setCount] = useState(0);  // トップレベル
  return <p>{count}</p>;
}

// NG
function Counter() {
  if (true) {
    const [count, setCount] = useState(0);  // if の中はダメ！
  }
}
```

なぜ？ React は `useState` が **呼ばれた順番** で state を管理しているからです。
条件分岐の中に入れると、呼ばれたり呼ばれなかったりして順番が狂います。

## まとめ

- **State** はコンポーネントの「記憶」（ホワイトボード）
- **普通の変数** を変えても React は気づかない → 画面が更新されない
- **useState** を使うと、値の変更を React に通知できる
- `useState` は `[値, 更新関数]` を返す
- 更新関数を呼ぶと React が **再レンダリング** する
- 各レンダリングは **スナップショット**（その時点の状態の写真）
