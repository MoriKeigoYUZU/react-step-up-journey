## 作るもの: ブックマークマネージャー

ウェブサイトのブックマーク（お気に入り）を管理するアプリです。

**機能一覧**:
- ブックマークの追加（タイトル + URL）
- ブックマークの削除
- お気に入り（★ / ☆）の切り替え
- 「すべて」「お気に入り」のフィルタリング

### アプリの構成

```
BookmarkApp（アプリ全体）
  ├── BookmarkForm（追加フォーム）
  ├── BookmarkFilter（フィルタ切り替え）
  └── BookmarkCard × N（各ブックマーク）
```

---

## 開発の進め方: テスト駆動で段階的に

このプロジェクトは3つの演習に分かれています。
**順番に進める** ことが大事です。

### ステップ1: データモデル（Exercise 01）

まず **データとロジック** を作ります。UI はまだ作りません。

```
「何を管理するか」「どんな操作ができるか」を決める
→ カスタムフック useBookmarks を作る
→ 中で useReducer を使って状態管理する
```

### アナロジー: 家を建てる

家を建てるとき、最初に壁紙や家具を選びますか？
いいえ、まず **基礎工事**（データモデル）をして、**骨組み**（ロジック）を作ってから、
**内装**（UI）を仕上げますよね。

ソフトウェア開発も同じです:

1. **基礎 = データモデル（型定義）**: `Bookmark` ってどんな形？
2. **骨組み = ロジック（カスタムフック）**: 追加・削除・切り替えの処理
3. **内装 = UI（コンポーネント）**: フォーム、カード、フィルター

### ステップ2: UIコンポーネント（Exercise 02）

次に **個々のUIパーツ** を作ります。

- `BookmarkForm`: タイトルとURLを入力するフォーム
- `BookmarkCard`: 1つのブックマークを表示するカード
- `BookmarkFilter`: フィルター切り替えボタン

この段階では、各コンポーネントは **独立してテスト** されます。
props として関数を受け取り、実際のデータとは接続しません。

### ステップ3: 統合（Exercise 03）

最後に、**ステップ1のフック** と **ステップ2のコンポーネント** を **組み合わせて** 完成させます。

```tsx
function BookmarkApp() {
  // ステップ1: データ管理フック
  const { bookmarks, addBookmark, removeBookmark, toggleFavorite } = useBookmarks();

  // フィルタ state
  const [filter, setFilter] = useState<'all' | 'favorites'>('all');

  // フィルタリング
  const filtered = filter === 'favorites'
    ? bookmarks.filter(b => b.favorite)
    : bookmarks;

  // ステップ2: UIコンポーネントを組み合わせる
  return (
    <div>
      <BookmarkForm onAdd={addBookmark} />
      <BookmarkFilter filter={filter} onFilterChange={setFilter} />
      {filtered.map(bookmark => (
        <BookmarkCard key={bookmark.id} ... />
      ))}
    </div>
  );
}
```

## 使う知識の整理

| 知識 | どこで使うか |
|------|-------------|
| **useReducer** | useBookmarks フック内で、ブックマークの追加・削除・切り替えを管理 |
| **カスタムフック** | useBookmarks として、ロジックをコンポーネントから分離 |
| **useState** | フォームの入力値、フィルター状態の管理 |
| **コンポーネント分割** | Form、Card、Filter を独立したコンポーネントに |
| **props** | 親から子へデータと関数を渡す |
| **イベント処理** | ボタンクリック、フォーム送信 |
| **リスト表示** | bookmarks.map() で一覧表示 |
| **条件分岐** | お気に入りフィルタ、★/☆ の切り替え |

## テスト駆動開発（TDD）の実践

この最終プロジェクトは、まさに **テスト駆動開発** の実践です。

1. **テストを読む**: 「何を作ればいいか」を理解する
2. **テストを実行**: 赤くなることを確認する（まだ何も実装していない）
3. **実装する**: テストが緑になるようにコードを書く
4. **リファクタリング**: テストが緑のまま、コードをきれいにする

実務でも、テストが「仕様書」として機能します。
「テストが求めるもの = 実装すべきもの」という考え方は、プロの現場でも大切にされています。
