# Chapter 02: コンポーネントとProps

## コンポーネントとは？

Reactでは、画面のパーツを **コンポーネント** という単位で作ります。

### レゴブロックで考えてみよう

レゴで家を作ることを想像してください。

- **壁ブロック** = `Wall` コンポーネント
- **屋根ブロック** = `Roof` コンポーネント
- **ドアブロック** = `Door` コンポーネント
- **家** = `House` コンポーネント（Wall + Roof + Door を組み合わせたもの）

コンポーネントは **再利用可能な部品** です。
一度作ったら、何度でも使えます。色違いのドアが欲しければ、同じ `Door` ブロックに「赤」と指定するだけ。

```tsx
// 「ボタン」というパーツを定義
function Button() {
  return <button>クリック</button>;
}

// 何度でも使える
function App() {
  return (
    <div>
      <Button />
      <Button />
      <Button />
    </div>
  );
}
```

## Propsとは？

レゴブロックの例をもう少し考えてみましょう。

同じ「ドアブロック」でも、**色** を変えたいときがありますよね。
「赤いドア」「青いドア」を作るために、ドアブロックに **指示** を渡す。

この **指示** が **Props** です。

### Propsは「親から子への手紙」

Props は **親コンポーネントから子コンポーネントに渡されるデータ** です。

「手紙」に例えると:
- 親が手紙を書いて子に渡す（データを送る）
- 子は手紙を **読むことはできる** が、**書き換えることはできない**（読み取り専用）
- 手紙の内容に応じて、子は自分の行動を変える（表示を変える）

```tsx
// 親コンポーネント（手紙を書く側）
function App() {
  return (
    <div>
      <Greeting name="太郎" />   {/* 「太郎」という手紙を渡す */}
      <Greeting name="花子" />   {/* 「花子」という手紙を渡す */}
    </div>
  );
}

// 子コンポーネント（手紙を受け取る側）
function Greeting(props: { name: string }) {
  return <h1>こんにちは、{props.name}さん！</h1>;
}
```

### なぜ一方通行なの？

Props が **親 → 子** の一方通行なのには理由があります。

想像してみてください。もし全員が好き勝手にデータを書き換えられたら？

- 「あれ、誰がこの値を変えたんだ？」
- 「いつ変わったのかわからない」
- 「バグの原因が追えない」

一方通行にすることで:
- **データの流れが予測可能** になる
- **どこで値が変わったか追跡しやすい**
- **デバッグが楽になる**

これを React では **単方向データフロー（one-way data flow）** と呼びます。

## TypeScript で Props の型を定義する

TypeScript を使うと、Props に **どんなデータを渡せるか** を明確に定義できます。

```tsx
// 方法1: type で定義
type GreetingProps = {
  name: string;
  age: number;
};

function Greeting(props: GreetingProps) {
  return (
    <div>
      <h1>{props.name}さん</h1>
      <p>{props.age}歳</p>
    </div>
  );
}

// 方法2: 分割代入を使う（よく使われる書き方）
function Greeting({ name, age }: GreetingProps) {
  return (
    <div>
      <h1>{name}さん</h1>
      <p>{age}歳</p>
    </div>
  );
}
```

型を定義しておくと:
- 間違った型のデータを渡そうとすると **エラーが出る**
- エディタの **自動補完が効く**
- **ドキュメント代わり** にもなる（このコンポーネントには何が必要か一目でわかる）

## コンポーネントを組み合わせて画面を作る

### 「家を建てる」ように

コンポーネントを組み合わせることで、複雑な画面を作れます。

```
App（家全体）
├── Header（屋根）
│   └── Logo, Navigation
├── Main（部屋）
│   ├── Sidebar（棚）
│   └── Content（家具）
└── Footer（基礎）
```

```tsx
function App() {
  return (
    <div>
      <Header />
      <main>
        <Sidebar />
        <Content />
      </main>
      <Footer />
    </div>
  );
}
```

各コンポーネントが **自分の役割だけを担当** する。
これにより:
- **コードが整理される**（1つのファイルが巨大にならない）
- **再利用しやすい**（Header は別のページでも使える）
- **修正が楽**（Header を変えたければ Header だけ修正すればいい）

## Props とローカル変数の違い

初学者がよく混同するのが「Props」と「コンポーネント内の変数」です。

```tsx
function ProductCard(props: { name: string; price: number }) {
  // これは Props（外から渡される）
  const { name, price } = props;

  // これはローカル変数（コンポーネント内で作る）
  const taxIncluded = price * 1.1;
  const isExpensive = price > 5000;

  return (
    <div>
      <h3>{name}</h3>
      <p>{price}円（税込 {taxIncluded}円）</p>
      {isExpensive && <span>高額商品</span>}
    </div>
  );
}
```

| | Props | ローカル変数 |
|---|---|---|
| どこから来る？ | 親コンポーネントから | コンポーネント内で定義 |
| 変更できる？ | できない（読み取り専用） | 自由に使える |
| 役割 | 外部からの設定 | 内部の計算や加工 |

## まとめ

- **コンポーネント** はUIの再利用可能な部品（レゴブロック）
- **Props** は親から子への一方通行のデータ（手紙）
- Props は **読み取り専用** で、子から変更できない
- TypeScript で型を定義すると安全で便利
- コンポーネントを **組み合わせて** 複雑な画面を構築する
